## Step 1a — Multi-week trend query

Run a single `run_select_query` covering up to 4 rolling 7-day windows to build a trend view. This powers the WoW delta line and trend table in the final report:

```sql
SELECT
  countIf(created_at >= now() - INTERVAL 7 DAY)                                                                    AS w0_total,
  countIf(created_at >= now() - INTERVAL 14 DAY AND created_at < now() - INTERVAL 7 DAY)                           AS w1_total,
  countIf(created_at >= now() - INTERVAL 21 DAY AND created_at < now() - INTERVAL 14 DAY)                          AS w2_total,
  countIf(created_at >= now() - INTERVAL 28 DAY AND created_at < now() - INTERVAL 21 DAY)                          AS w3_total,
  round(avgIf(resolution_minutes, created_at >= now() - INTERVAL 7 DAY))                                           AS w0_mttr,
  round(avgIf(resolution_minutes, created_at >= now() - INTERVAL 14 DAY AND created_at < now() - INTERVAL 7 DAY))  AS w1_mttr,
  round(avgIf(resolution_minutes, created_at >= now() - INTERVAL 21 DAY AND created_at < now() - INTERVAL 14 DAY)) AS w2_mttr,
  round(avgIf(resolution_minutes, created_at >= now() - INTERVAL 28 DAY AND created_at < now() - INTERVAL 21 DAY)) AS w3_mttr,
  round(sumIf(cost_usd, created_at >= now() - INTERVAL 7 DAY), 2)                                                  AS w0_cost,
  round(sumIf(cost_usd, created_at >= now() - INTERVAL 14 DAY AND created_at < now() - INTERVAL 7 DAY), 2)         AS w1_cost,
  round(sumIf(cost_usd, created_at >= now() - INTERVAL 21 DAY AND created_at < now() - INTERVAL 14 DAY), 2)        AS w2_cost,
  round(sumIf(cost_usd, created_at >= now() - INTERVAL 28 DAY AND created_at < now() - INTERVAL 21 DAY), 2)        AS w3_cost,
  countIf(is_partner_impacting = 1 AND created_at >= now() - INTERVAL 7 DAY)                                       AS w0_partner,
  countIf(is_partner_impacting = 1 AND created_at >= now() - INTERVAL 14 DAY AND created_at < now() - INTERVAL 7 DAY)  AS w1_partner,
  countIf(is_partner_impacting = 1 AND created_at >= now() - INTERVAL 21 DAY AND created_at < now() - INTERVAL 14 DAY) AS w2_partner,
  countIf(is_partner_impacting = 1 AND created_at >= now() - INTERVAL 28 DAY AND created_at < now() - INTERVAL 21 DAY) AS w3_partner
FROM serviceportal.incident_reports
WHERE created_at >= now() - INTERVAL 28 DAY
```

From the result, compute:
- `wow_total_delta` = w0_total - w1_total (and as percentage: `abs(round((w0_total - w1_total) / w1_total * 100))`% — always unsigned; the ↓/↑ arrow carries the sign)
- `wow_mttr_delta` = w0_mttr - w1_mttr
- `wow_cost_delta` = w0_cost - w1_cost
- `wow_partner_delta` = w0_partner - w1_partner

Only include w2/w3 in the trend table if their total is > 0 — they will be zero until ~mid/late May as the table accumulates history.

## Step 2 — Identify the most frequent incident patterns

Call `run_select_query` with the following SQL to find the most repeated incident types. Numbers in titles are normalised to `N` so that e.g. "16 messages" and "83 messages" are treated as the same alert. Results are ordered by **total AI investigation cost** (highest spend first) so that the most wasteful patterns surface at the top:

```sql
SELECT
  replaceRegexpAll(title, '[0-9]+', 'N') AS alert_pattern,
  count() AS occurrences,
  countIf(is_partner_impacting = 1) AS partner_impacts,
  round(avg(resolution_minutes)) AS avg_resolve_min,
  round(min(resolution_minutes)) AS min_resolve_min,
  round(sum(cost_usd), 2) AS total_ai_cost_usd,
  round(avg(cost_usd), 3) AS avg_ai_cost_usd,
  max(toDate(created_at)) AS last_seen
FROM serviceportal.incident_reports
WHERE created_at >= now() - INTERVAL 7 DAY
GROUP BY alert_pattern
HAVING occurrences >= 3
ORDER BY total_ai_cost_usd DESC
LIMIT 25
```

Review the results and note the top 10 patterns by cost. For each, observe:
- How many times per day it fires on average (occurrences ÷ 7)
- Whether it ever has partner impact
- Whether the minimum resolution time is very short (< 5 minutes) — this signals self-healing, i.e. pure noise
- Whether `total_ai_cost_usd` is high relative to occurrences — this flags patterns that are expensive to investigate but provide no operational value

## Step 2a — Detect duplicate investigations

Call `run_select_query` to find incidents investigated twice in the same day — once via PagerDuty and once via incident.io for the same physical event:

```sql
SELECT
  replaceRegexpAll(title, '[0-9]+', 'N') AS alert_pattern,
  toDate(started_at) AS incident_date,
  count() AS investigation_count,
  round(sum(cost_usd), 3) AS total_cost_usd,
  groupArray(source) AS sources,
  countIf(source = 'pagerduty') AS pd_count,
  countIf(source = 'incident_io') AS incidentio_count
FROM serviceportal.incident_reports
WHERE created_at >= now() - INTERVAL 7 DAY
GROUP BY alert_pattern, incident_date
HAVING pd_count > 0 AND incidentio_count > 0
ORDER BY total_cost_usd DESC
LIMIT 10
```

Only rows where **both** PagerDuty and incident.io investigated the same event qualify as duplicates — this prevents multiple fires of the same alert on the same day from being miscounted. Sum `total_cost_usd` across all returned rows to get `dupe_total_cost_usd`. Record the row count as `dupe_count` (number of distinct events with cross-source duplicate investigations) for the Step 7 headline.

## Step 2b — Detect MTTR regression

Call `run_select_query` to compare average resolution time this week vs the prior week per alert pattern. Patterns where TTR has increased by more than 5 minutes are getting harder to resolve — early signal of a worsening issue:

```sql
SELECT
  replaceRegexpAll(title, '[0-9]+', 'N') AS alert_pattern,
  round(avgIf(resolution_minutes, created_at >= now() - INTERVAL 7 DAY)) AS this_week_avg,
  round(avgIf(resolution_minutes, created_at < now() - INTERVAL 7 DAY
    AND created_at >= now() - INTERVAL 14 DAY)) AS prior_week_avg,
  round(
    avgIf(resolution_minutes, created_at >= now() - INTERVAL 7 DAY)
    - avgIf(resolution_minutes, created_at < now() - INTERVAL 7 DAY
      AND created_at >= now() - INTERVAL 14 DAY)
  ) AS delta_min
FROM serviceportal.incident_reports
WHERE created_at >= now() - INTERVAL 14 DAY
GROUP BY alert_pattern
HAVING this_week_avg > 0 AND prior_week_avg > 0 AND delta_min > 5
ORDER BY delta_min DESC
LIMIT 10
```

Flag any result as a MTTR regression candidate. These feed into the RELIABILITY GAPS section in Step 7.

## Step 2c — Detect carryover patterns

Find alert patterns that appear in **both** this week's top-10 cost patterns **and** last week's top-10. These are patterns that were flagged last week and haven't been resolved — the clearest signal that the sprint backlog isn't being actioned:

```sql
WITH this_week AS (
  SELECT
    replaceRegexpAll(title, '[0-9]+', 'N') AS alert_pattern,
    count() AS occurrences,
    round(sum(cost_usd), 2) AS total_cost
  FROM serviceportal.incident_reports
  WHERE created_at >= now() - INTERVAL 7 DAY
  GROUP BY alert_pattern
  HAVING occurrences >= 3
  ORDER BY total_cost DESC
  LIMIT 10
),
prior_week AS (
  SELECT
    replaceRegexpAll(title, '[0-9]+', 'N') AS alert_pattern,
    count() AS prior_occurrences,
    round(sum(cost_usd), 2) AS prior_cost
  FROM serviceportal.incident_reports
  WHERE created_at >= now() - INTERVAL 14 DAY
    AND created_at < now() - INTERVAL 7 DAY
  GROUP BY alert_pattern
  HAVING prior_occurrences >= 3
  ORDER BY prior_cost DESC
  LIMIT 10
)
SELECT
  tw.alert_pattern,
  tw.total_cost,
  tw.occurrences,
  pw.prior_occurrences,
  pw.prior_cost
FROM this_week tw
INNER JOIN prior_week pw ON tw.alert_pattern = pw.alert_pattern
ORDER BY tw.total_cost DESC
```

Record the returned rows as **carryover items**. Set `carryover_count` = number of rows returned. These patterns will receive a `📌 Week 2+:` prefix in the Step 7 sprint backlog.

## Step 5 — Synthesise findings through an SRE lens

Using the data from Steps 1–4, produce a categorised assessment. Apply these principles:

**Toil** = Work that is repetitive, reactive, and devoid of enduring value. Every incident that fires on the same pattern repeatedly and self-heals is toil. The SRE goal is to eliminate the toil, not manage it.

Organise findings into four categories:

### A — Top Toil Sources
Patterns that fire 3+ times per week, consistently self-resolve quickly, and have low or zero partner impact. These are noise: the underlying issue needs fixing, the alert threshold needs tuning, or an inhibition rule should suppress the alert. For each, note frequency, partner impact count, and the specific recommended fix drawn from the sampled report's Next Steps.

### B — Reliability Gaps
Patterns that recur AND carry partner impact, or single high-severity incidents with long MTTR. These represent systemic failure modes: scaling limits, missing capacity, architecture weaknesses, or absent runbooks. For each, note the root cause (from Findings), partner impact, and the fix (from Next Steps).

### C — Observability Gaps
Complaints about missing metrics, logs, or runbooks appearing across the sampled reports from Steps 4 and 4a (up to 20 reports total). If the same gap appears in 2 or more reports — even for different brands or services — it is high priority. Aggregate and deduplicate across all sampled reports. For each gap, note how many reports cite it and the specific recommendation (from Observability Improvements). Do not include threshold or alerting tuning suggestions here — those belong in Category D.

### D — Alerting Quality Issues
Alerts that fire on transient conditions that always self-resolve within minutes. Distinct from Category A: here the service is healthy, but the alerting is too sensitive. These need threshold tuning, grace periods, or hold-down rules.

## Step 6 — Query brand/service breakdown

Call `run_select_query` to get the per-brand incident count for the thread reply table. The WHERE clause covers 14 days so that the prior-week counts are available for the Δ WoW column:

```sql
SELECT
  CASE
    WHEN title REGEXP '^[A-Za-z][A-Za-z0-9][A-Za-z0-9 _-]*:'
      THEN lower(trimBoth(extract(title, '^([^:]+):')))
    WHEN title REGEXP '^[a-z][a-z0-9-]+ - '
      THEN lower(trimBoth(extract(title, '^([a-z][a-z0-9-]+) - ')))
    WHEN title LIKE '[FIRING%'
      THEN 'Infrastructure (AWS/Grafana)'
    WHEN title LIKE 'ftcrm%'
      THEN 'ftcrm'
    ELSE 'Other'
  END AS brand_service,
  countIf(created_at >= now() - INTERVAL 7 DAY)                                            AS incidents,
  countIf(created_at >= now() - INTERVAL 14 DAY AND created_at < now() - INTERVAL 7 DAY)  AS incidents_prev,
  countIf(severity = 'P2' AND created_at >= now() - INTERVAL 7 DAY)                       AS p2,
  countIf(severity = 'P3' AND created_at >= now() - INTERVAL 7 DAY)                       AS p3,
  countIf(is_partner_impacting = 1 AND created_at >= now() - INTERVAL 7 DAY)              AS partner_impacting,
  countIf(status = 'escalated_to_soc' AND created_at >= now() - INTERVAL 7 DAY)           AS escalated,
  round(avgIf(resolution_minutes, created_at >= now() - INTERVAL 7 DAY))                  AS avg_mttr_min
FROM serviceportal.incident_reports
WHERE created_at >= now() - INTERVAL 14 DAY
GROUP BY brand_service
ORDER BY incidents DESC
LIMIT 35
```

For each row, compute `delta = incidents - incidents_prev` for use in the Δ WoW column of the thread reply table.
